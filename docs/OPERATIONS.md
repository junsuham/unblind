# 언블라인드 운영 절차

## 매일 확인

1. `/admin`에서 미처리 신고, 24시간 경과 신고, 가입 승인 대기를 확인한다.
2. `/admin/health`에서 치명적 오류와 오류 증가 여부를 확인한다.
3. 숨김 콘텐츠가 자동 숨김인지 운영자 조치인지 감사 이력과 함께 확인한다.
4. Vercel 배포 상태와 `/api/health`의 데이터베이스 연결 상태를 확인한다.

## 신고 대응 기준

- 생명 위험, 개인정보 노출, 금전 피해 가능성이 있으면 최우선으로 검토한다.
- 숨김·삭제·복구·문제 없음 조치에는 3자 이상의 구체적인 메모를 남긴다.
- 신고자와 작성자 이메일은 안전 운영 목적 외에는 열람하거나 외부 공유하지 않는다.
- 조치에 이의가 접수되면 최초 조치자가 아닌 다른 관리자가 재검토하는 것을 권장한다.

## 장애 대응

1. 사용자 영향 범위와 최초 발생 시각을 기록한다.
2. `/api/health`, Vercel 함수 오류, Supabase 상태, 최근 배포를 순서대로 확인한다.
3. 데이터 손상 가능성이 있으면 쓰기 기능을 먼저 제한하고 최근 정상 배포로 되돌린다.
4. 인증 비밀이나 관리자 키 노출이 의심되면 즉시 키를 교체하고 관련 세션을 폐기한다.
5. 복구 후 원인, 영향, 대응, 재발 방지 조치를 운영 조치 이력에 남긴다.

## 백업과 복구

- Supabase 프로젝트의 백업 제공 범위와 보존 기간을 요금제 화면에서 매월 확인한다.
- 스키마 변경 전 데이터베이스 백업 또는 별도 내보내기를 확보한다.
- 월 1회 비운영 환경에서 복원 절차를 연습하고, 게시글·댓글·신고·승인 목록의 건수를 대조한다.
- Storage의 비공개 `post-images` 버킷도 데이터베이스와 함께 복구 범위에 포함한다.
- 복구 작업 중에는 서비스 쓰기를 중단해 백업 시점 이후 데이터가 섞이지 않도록 한다.

## 비밀정보 교체

- `SUPABASE_SECRET_KEY`, `ADMIN_SESSION_TOKEN`, `ADMIN_PASSWORD`, `CRON_SECRET`, `VAPID_PRIVATE_KEY`는 서버 환경 변수에만 저장한다.
- 담당자 변경, 노출 의심, 외부 공유 발생 시 즉시 교체한다.
- 교체 후 로그인, 관리자 요청, 푸시 발송, 계정 삭제를 각각 한 번씩 점검한다.

## 정기 정리

알림 발송 작업은 오류 기록 90일, 요청 제한 기록 2일, 읽은 알림 90일, 비활성 푸시 구독 30일, 삭제 감사 기록 1년, 운영 조치 2년 기준의 정리 작업도 함께 실행한다.
