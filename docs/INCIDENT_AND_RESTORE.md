# 장애 대응 및 복구 절차

## 목표

- 사용자 안전, 계정·게시물 무결성, 개인정보 보호를 우선한다.
- 목표 RPO는 24시간 이내, 목표 RTO는 4시간 이내다.
- 실제 Supabase 요금제와 백업 보존 범위가 더 짧으면 공개 전 상향한다.

## 장애 판단과 초기 대응

1. `/api/health`, Vercel 배포 상태, Supabase 상태와 최근 오류 이벤트를 확인한다.
2. 개인정보 노출이나 데이터 손상이 의심되면 쓰기 기능을 먼저 제한하고 증거를 보존한다.
3. 최근 배포가 원인이면 Vercel에서 직전 정상 배포로 롤백한다.
4. 인증·데이터베이스 장애면 Supabase 상태를 확인하고 복구 시점까지 사용자에게 점검 상태를 안내한다.

## 데이터 복구

1. 현재 데이터베이스와 파일 저장소를 별도 백업하고 복구 대상 시점을 기록한다.
2. Supabase 백업 또는 PITR에서 별도 검증 프로젝트로 먼저 복구한다.
3. 마이그레이션 버전, 사용자 수, 최근 게시글·댓글·신고·문의 건수와 참조 무결성을 비교한다.
4. 개인정보 접근 로그와 운영 조치 이력의 연속성을 확인한다.
5. 검증 프로젝트가 정상일 때만 운영 연결 정보를 전환한다.

## 복구 검증

- Google 로그인, 승인 사용자 판정, 글·댓글 읽기와 쓰기, 신고·차단, 계정 삭제를 확인한다.
- 고객지원 접수와 관리자 처리, 알림, YouTube 재생, 공개 정책 페이지를 확인한다.
- `/api/health`와 핵심 화면의 오류율·응답 시간을 30분 이상 관찰한다.
- 문제가 남으면 전환을 취소하고 직전 정상 상태로 돌아간다.

## 커뮤니케이션과 사후 조치

- 영향 범위, 시작·복구 시각, 임시 조치와 사용자 행동 필요 여부를 앱 공지에 기록한다.
- 개인정보 침해가 확인되면 관련 법령과 보호위원회 지침에 따라 통지·신고한다.
- 원인, 재발 방지 항목, 담당자와 기한을 장애 기록에 남긴다.

## 분기별 복구 훈련

- 분기마다 비운영 프로젝트에서 백업 복구와 핵심 흐름 검증을 수행한다.
- 실제 RPO·RTO, 누락 데이터, 실패 단계와 개선 항목을 기록한다.
- 복구 훈련 없이 운영 데이터베이스를 직접 복원하지 않는다.
